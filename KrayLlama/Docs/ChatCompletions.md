# Endpoint `/v1/chat/completions`

Эндпоинт `/v1/chat/completions` в Ollama реализован для совместимости с API OpenAI. Это удобно, если вы переходите с сервисов OpenAI на локальные модели или используете инструменты, которые «заточены» под стандарт GPT.

Вот основные параметры, которые можно передать в теле JSON-запроса:

## 1. Обязательные параметры

* `model`: Название модели, которую вы хотите использовать (например, "llama3", "mistral").
* `messages`: Список объектов сообщений, представляющих историю диалога. Каждый объект содержит:
  * `role`: роль отправителя ("system", "user", "assistant").
  * `content`: текст сообщения.

## 2. Общие параметры управления

* `stream`: Булево значение (true по умолчанию). Если false, Ollama вернет ответ целиком одним объектом, когда генерация завершится.
* `max_tokens`: Максимальное количество токенов, которое модель может сгенерировать в ответе.
* `temperature`: Степень случайности ответа (от 0 до 2). Чем выше значение, тем «творческим» и непредсказуемым будет ответ. При 0 модель становится максимально детерминированной.
* `top_p`: Альтернатива температуре (ядерное сэмплирование). Модель рассматривает только те токены, общая вероятность которых составляет p.
* `seed`: Если указано, система постарается генерировать детерминированные результаты (одинаковые ответы на одинаковые запросы).

## 3. Специфические опции Ollama (options)

Хотя этот эндпоинт эмулирует OpenAI, Ollama позволяет передавать дополнительные низкоуровневые параметры через поле options. Это важная особенность для тонкой настройки:

* num_ctx: Размер контекстного окна (например, 4096).
* num_predict: Максимальное количество токенов для генерации (аналог max_tokens).
* top_k: Ограничивает выборку только k самыми вероятными токенами.
* repeat_penalty: Штраф за повторение слов (например, 1.1).
* stop: Список стоп-последовательностей (строк), при появлении которых модель прекратит генерацию.

## Пример запроса (CURL):

```bash
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama3",
    "messages": [
      {"role": "system", "content": "Ты полезный ассистент."},
      {"role": "user", "content": "Привет! Как дела?"}
    ],
    "stream": false,
    "temperature": 0.7,
    "options": {
      "num_ctx": 2048,
      "repeat_penalty": 1.2
    }
  }'
```

Важный нюанс: Параметры внутри объекта `options` соответствуют тем, что используются в нативном Ollama API (`/api/chat`). Если вы используете стандартные библиотеки OpenAI (на Python или JS), они могут не знать про поле options, поэтому его часто приходится передавать в `extra_body`.

В отличие от проприетарных API (например, OpenAI), у Ollama нет специального технического поля `refusal` в JSON-ответе.

Ollama работает как «прозрачный» мост к локальной модели. Это означает, что если модель решает отклонить запрос (из-за встроенных цензурных фильтров или настроек безопасности), она делает это как обычный ответ.

Вот как это выглядит на практике для эндпоинта /v1/chat/completions:

## 1. Как выглядит отказ (Refusal)

```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1715000000,
  "model": "llama3",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Извините, я не могу выполнить этот запрос, так как он нарушает..." 
      },
      "finish_reason": "stop"
    }
  ]
}
```

* `finish_reason`: Будет `stop`, так как модель успешно сгенерировала текст (текст отказа).
* `refusal`: Это поле (введенное в OpenAI API для GPT-4o) в Ollama на данный момент отсутствует.

## 2. Технические ошибки (не путать с отказом модели)

Если же произошла именно системная ошибка (модель не найдена, не хватило памяти), Ollama вернет соответствующий HTTP-статус и JSON с полем error:

* 404 Not Found: Модель не загружена.
* 500 Internal Server Error: Ошибка при генерации.
* JSON-структура ошибки:

```json
{ "error": "model 'llama4' not found" }
```

## 3. Как программно определить отказ?

Поскольку Ollama не помечает отказы технически, разработчикам приходится использовать косвенные методы:

1. Семантический анализ: Поиск ключевых слов в начале ответа (например, "I cannot assistance", "Извините, я не могу").

2. Классификация: Использование более легкой модели для проверки, является ли ответ основной модели отказом.

3. Логиты (сложный путь): Некоторые разработчики измеряют вероятность (logits) первых токенов. Если вероятность слов типа «Sorry» или «No» выше, чем «Sure», это считается отказом.

Итог: Для Ollama отказ — это просто текстовый ответ модели. Вам нужно обрабатывать его самостоятельно в коде вашего приложения.
