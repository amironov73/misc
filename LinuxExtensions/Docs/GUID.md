# GUID в Linux

Для Linux-систем для работы с GUID стандартом де-факто является библиотека `libuuid`. Она входит в состав пакета `util-linux` и почти наверняка уже установлена у вас в системе.
Вот пошаговый план, как это реализовать:

## 1. Установка библиотеки

Если вы используете Ubuntu/Debian, установите пакет для разработки:

```bash
sudo apt-get install uuid-dev
```

## 2. Код на C

Библиотека предоставляет тип `uuid_t` и функции для генерации и преобразования GUID в строку.

```c++
#include <stdio.h>
#include <uuid/uuid.h>

int main() {
    uuid_t binuuid;
    // Резервируем 37 байт: 36 для UUID + 1 для нулевого символа '\0'
    char uuid_str[37];

    // 1. Генерируем случайный UUID (версия 4)
    uuid_generate_random (binuuid);

    // 2. Преобразуем бинарный формат в читаемую строку (нижний регистр)
    uuid_unparse_lower (binuuid, uuid_str);

    printf ("Сгенерированный UUID: %s\n", uuid_str);

    return 0;
}
```

## 3. Компиляция

При сборке обязательно нужно линковать библиотеку флагом `-luuid`:

```bash
gcc main.c -o my_app -luuid
```

## Разница между функциями генерации:

* `uuid_generate()` — пытается использовать качественный генератор случайных чисел (`/dev/urandom`), а если его нет, берет идентификатор MAC-адреса и время.

* `uuid_generate_random()` — заставляет использовать исключительно случайные числа (UUID v4). Для имен файлов это обычно лучший выбор.

* `uuid_generate_time()` — использует время и MAC-адрес (UUID v1). Это может быть небезопасно с точки зрения конфиденциальности, так как по UUID можно вычислить, на какой машине и когда создан файл.

## Альтернатива (без библиотек)

Если вы не хотите тянуть лишние зависимости, в Linux можно просто прочитать готовый GUID прямо из ядра:

```c++
FILE *f = fopen ("/proc/sys/kernel/random/uuid", "r");
```

Но программный метод через `libuuid` считается более "чистым" и надежным для серьезных расширений.

Если вы планируете создавать пару связанных файлов (входной и выходной), использование одного и того же GUID для обоих имен — отличное решение. Это позволит вам легко находить их в файловой системе по общему идентификатору.

## Пример кода

```c++
#include <stdio.h>
#include <uuid/uuid.h>

int main() {
    uuid_t binuuid;
    char uuid_str[37];
    char input_filename[64];
    char output_filename[64];

    // 1. Генерируем один уникальный ID для пары файлов
    uuid_generate_random(binuuid);
    uuid_unparse_lower(binuuid, uuid_str);

    // 2. Формируем имена файлов с использованием общего UUID
    snprintf(input_filename, sizeof(input_filename), "input_%s.txt", uuid_str);
    snprintf(output_filename, sizeof(output_filename), "output_%s.txt", uuid_str);

    // 3. Создаем файлы (пример)
    FILE *f_in = fopen(input_filename, "w");
    if (f_in) {
        fprintf(f_in, "Данные для обработки\n");
        fclose(f_in);
        printf("Создан файл: %s\n", input_filename);
    }

    FILE *f_out = fopen(output_filename, "w");
    if (f_out) {
        fprintf(f_out, "Результаты обработки\n");
        fclose(f_out);
        printf("Создан файл: %s\n", output_filename);
    }

    return 0;
}
```

## Важные детали реализации:

1. Размер буфера: `UUID` в строковом представлении всегда занимает 36 символов + 1 байт на завершающий ноль (\0). Буферы `input_filename` и `output_filename` должны быть достаточно большими, чтобы вместить префикс, `UUID` и расширение (в примере взято 64 байта с запасом).

2. Атомарность: Если ваше расширение работает в многопоточной среде или под высокой нагрузкой, помните, что `fopen(..., "w")` перезапишет файл, если он уже существует. Хотя шанс коллизии `GUID` ничтожно мал, для максимальной безопасности иногда используют системный вызов `open` с флагами `O_CREAT | O_EXCL`, который вернет ошибку, если файл с таким именем уже есть.

3. Путь к файлам: В примере файлы создаются в текущей рабочей директории. Если вам нужно создавать их во временной папке (например, `/tmp`), не забудьте добавить путь в `snprintf`.

## Временные файлы

Для обмена данными между программами в Linux лучше всего использовать директорию `/tmp`. Вот основные причины и механизмы работы с ней:

### 1. Где создавать файлы?

Стандартное место — `/tmp`. В современных дистрибутивах Linux (основанных на systemd) содержимое `/tmp` часто хранится в оперативной памяти (tmpfs). Это делает чтение и запись файлов между вашим расширением и внешней программой очень быстрыми.

Как правильно получить путь:

* Вместо того чтобы жестко прописывать `/tmp/`, лучше использовать переменную окружения TMPDIR. Если она не задана, тогда откатываться к `/tmp`.

```c++
const char *tmp_dir = getenv ("TMPDIR");
if (tmp_dir == NULL) {
    tmp_dir = "/tmp";
}
// Формируем полный путь: snprintf(path, sizeof(path), "%s/input_%s.txt", tmp_dir, uuid_str);
```

## 2. Механизм удаления

В C есть два основных способа:

### Способ А: Функция remove() (Самый простой)

Это стандартная функция C (из stdio.h), которая удаляет файл по его имени.

```c++
if (remove (input_filename) == 0) {
    // Успешно удалено
} else {
    perror ("Ошибка при удалении входного файла");
}
```

### Способ Б: Системный вызов unlink()

Это низкоуровневый системный вызов POSIX (нужен `<unistd.h>`). Он делает ровно то же самое — удаляет имя файла из файловой системы.

## 3. Когда именно удалять?

Так как вы запускаете внешнюю программу, вам нужно дождаться её завершения, прежде чем чистить файлы. Типичная схема выглядит так:

1. Генерируете `UUID`.
2. Создаете `input_UUID.txt` и записываете данные.
3. Формируете командную строку для запуска.
4. Запускаете программу (например, через `system()` или связку `fork() + exec()`).
5. Дожидаетесь завершения внешней программы.
6. Читаете результат из `output_UUID.txt`.
7. Сразу после этого вызываете `remove()` для обоих файлов.

### Нюанс с автоматической очисткой

Если ваше расширение вдруг "упадет" и не успеет вызвать `remove()`, файлы останутся в `/tmp`. В Linux существует механизм очистки:

* Система автоматически удаляет старые файлы из `/tmp` (обычно те, к которым не обращались более 10–30 дней).

* При перезагрузке компьютера содержимое `/tmp` (если это `tmpfs`) очищается полностью.

Совет по безопасности:

Поскольку `/tmp` — общая папка для всех пользователей, убедитесь, что создаете файлы с правами доступа только для своего пользователя. Функция `fopen` создает файлы с учетом `umask`, но если нужна максимальная приватность, используйте chmod(filename, 0600) сразу после создания.

## Запуск внешней программы

Выбор между `system()` и `fork() + exec()` зависит от того, насколько тесно ваше приложение должно контролировать процесс выполнения.

Вот разбор, какой способ лучше подойдет для задачи «создал файлы — запустил программу — удалил файлы».

### 1. Способ system() — Простой, но рискованный

Функция `system("external-program in.txt out.txt")` удобна тем, что она блокирует выполнение вашей программы до тех пор, пока внешняя утилита не завершит работу.

* Плюсы: Минимум кода (одна строка).
* Минусы:
    * Безопасность: Она запускает командную оболочку (`/bin/sh`). Если в имени файла или `UUID` окажутся спецсимволы (хотя в `UUID` их нет), это может привести к инъекции команд.
    * Производительность: Лишний процесс (оболочка) тратит ресурсы.
    * Гибкость: Вы не можете легко перехватить ошибки запуска самой программы (например, если файл программы не найден).

### 2. Способ fork() + execvp() — Профессиональный и надежный

Для расширений Linux-приложений этот способ считается предпочтительным. Вы напрямую просите ядро создать процесс и заменить его на нужную программу. Почему это лучше для вас:

1. Безопасность: Аргументы передаются как массив, оболочка не вызывается, интерпретация символов невозможна.

2. Контроль: Вы точно знаете, запустилась ли программа, и можете получить её точный код возврата.

### Пример реализации:

```c++
#include <sys/wait.h>
#include <unistd.h>
#include <stdlib.h>
#include <stdio.h>

int run_external_program (const char *in, const char *out) {
    pid_t pid = fork();

    if (pid == -1) {
        perror ("fork failed");
        return -1;
    }

    if (pid == 0) {
        // Это дочерний процесс
        // Вызываем программу. Аргументы: путь, arg0, arg1, arg2, NULL
        execlp ("external-program", "external-program", in, out, (char *) NULL);
        
        // Если execlp вернул управление — значит произошла ошибка
        perror ("execlp failed");
        exit (EXIT_FAILURE);
    } else {
        // Это родительский процесс (ваше расширение)
        int status;
        waitpid (pid, &status, 0); // Ждем завершения именно нашей программы

        if (WIFEXITED (status)) {
            return WEXITSTATUS (status); // Возвращаем код завершения (0 = успех)
        }
    }
    return -1;
}
```

### Итоговая схема работы расширения:

1. Генерируете `UUID` через `uuid_generate_random`.
2. Формируете пути в `/tmp/input_UUID.txt` и `/tmp/output_UUID.txt`.
3. Пишете данные во входной файл.
4. Вызываете `run_external_program(in_path, out_path)`.
5. В родительском процессе: после возврата из функции проверяете код ошибки.
6. Читаете выходной файл.
7. Обязательно вызываете `remove(in_path)` и `remove(out_path)`.

### Когда все же выбрать system()?

Если ваше расширение — это маленький скриптоподобный инструмент «для себя», и вы на 100% доверяете строке команды, `system()` сэкономит вам 20 строк кода. Но для публичного расширения или плагина к серьезному приложению всегда используйте `fork + exec`.

С учетом того, что вам нужно только контролировать факт успешного запуска и код завершения, связка fork() + exec() + waitpid() — идеальный выбор. Она дает вам полный контроль над жизненным циклом процесса без лишних "прослоек" вроде командной оболочки.
Вот готовая логика для вашего расширения, которая обрабатывает оба сценария: если программу не удалось найти/запустить и если она завершилась с ошибкой.

### Реализация функции запуска

```c++
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <errno.h>

int run_process (const char *in_file, const char *out_file) {
    pid_t pid = fork();

    if (pid < 0) {
        perror ("Ошибка при создании процесса (fork)");
        return -1;
    }

    if (pid == 0) {
        // Дочерний процесс: заменяем себя внешней программой
        // execlp ищет программу в PATH. Если путь фиксированный, используйте execl
        execlp ("external-program", "external-program", in_file, out_file, (char *)NULL);

        // Если exec вернул управление — значит запуск не удался (например, файл не найден)
        perror ("Ошибка запуска внешней программы (exec)");
        exit (127); // Стандартный код ошибки "command not found"
    } else {
        // Родительский процесс: ждем завершения
        int status;
        if (waitpid (pid, &status, 0) == -1) {
            perror ("Ошибка при ожидании процесса (waitpid)");
            return -1;
        }

        if (WIFEXITED (status)) {
            int exit_code = WEXITSTATUS (status);
            if (exit_code != 0) {
                fprintf (stderr, "Программа завершилась с ошибкой. Код: %d\n", exit_code);
                return exit_code;
            }
            return 0; // Все отлично
        } else if (WIFSIGNALED (status)) {
            fprintf (stderr, "Программа была убита сигналом: %d\n", WTERMSIG (status));
            return -1;
        }
    }
    return -1;
}
```

### Как это работает:

1. Проверка запуска: Если `execlp` не найдет файл `external-program`, дочерний процесс завершится с кодом 127. Ваша основная программа увидит, что код не 0, и вы сможете сообщить пользователю: "Программа не установлена или недоступна".

2. Проверка "падения": Если программа упала (например, Segmentation Fault), макрос `WIFSIGNALED` поймает это, и вы поймете, что выходной файл может быть поврежден или не создан.

3. Безопасное удаление: После выхода из этой функции вы в любом случае вызываете `remove()` для обоих файлов.

## Маленький совет по надежности

Иногда внешние программы могут "зависать". Если это критично для вашего расширения, можно добавить простейший тайм-аут, используя `alarm()` перед `waitpid()`, но для большинства расширений стандартного ожидания вполне достаточно.
