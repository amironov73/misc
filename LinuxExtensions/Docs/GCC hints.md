# GCC hints

Допустим, у нас есть динамическая библиотека в формате DLL (под Windows) с исходными текстами на языке C. Она экспортировала функции, которые расширяли функциональность ИРБИС64. 

Теперь нам нужно портировать её под Linux. Если с самим алгоритмом проблем никаких нет - он не зависит от операционной системы, то с экспортом функций возникла проблема - я не знаю, как это правильно делается под Linux. В Windows (Microsoft Visual C++) функции экспортировались так:

```c++
#pragma comment(linker, "/export:WriteSummary=_WriteSummaryText@12")
int __stdcall WriteSummaryText
    (
        const char *inputBuffer,
        char *outputBuffer,
        int outputBufferSize
    )
{
    ...
}
```

Рассмотрим на примере GCC 8.3.

В Linux (и GCC в частности) нет прямого аналога `#pragma comment(linker, ...)`, так как формат исполняемых файлов ELF работает иначе, чем Windows PE. Для решения вашей задачи в GCC используется комбинация атрибутов видимости и Linker Script (или флаги линковщика).

1. Управление видимостью в коде

По умолчанию в Linux все функции считаются "экспортируемыми". Чтобы имитировать поведение Windows, рекомендуется компилировать проект с флагом `-fvisibility=hidden`, а нужные функции помечать явно:

```c++
#define DLL_EXPORT __attribute__((visibility("default")))

DLL_EXPORT int WriteSummaryText
    (
        const char *inputBuffer,
        char *outputBuffer, 
        int outputBufferSize
    )
{
    ...
}
```

2. Создание алиаса (аналог /export:NewName=OldName)
   
Чтобы функция `WriteSummaryText` была доступна под коротким именем `WriteSummary`, используйте атрибут `alias`:

```c++
DLL_EXPORT int WriteSummaryText
    (
        const char *inputBuffer, 
        char *outputBuffer, 
        int outputBufferSize
    ) 
{
    // реализация
}

// Создаем алиас
void WriteSummary() __attribute__((alias("WriteSummaryText")));
```

3. Использование Version Script (самый правильный метод)
   
Если вы хотите иметь полный контроль над именами (как в `.def` файле), создайте текстовый файл, например `exports.map`:

```
{
  global:
    WriteSummary = WriteSummaryText;
  local: *;
};
```

И добавьте его при линковке:

```bash
gcc -shared -o libsummary.so summary.o -Wl,--version-script=exports.map
```

Важный нюанс: в Linux нет "декорации" имен (вроде @12), характерной для 32-битного Windows `stdcall`. Имя в таблице экспорта будет ровно таким, какое вы укажете.

В Linux управление соглашениями о вызовах в GCC зависит от целевой архитектуры (`x86` vs `x86_64`), так как современный 64-битный Linux практически полностью отказался от многообразия соглашений в пользу единого стандарта.

1. Архитектура x86 (32 бита)

На 32-битном x86 вы действительно можете использовать различные соглашения. Для этого применяется атрибут `__attribute__((имя_соглашения))`:

* cdecl (стандартный C): `__attribute__((cdecl))` — аргументы в стеке, очищает вызывающий (caller).

* stdcall (Pascal-like): `__attribute__((stdcall))` — аргументы в стеке, очищает вызываемая функция (callee).

* fastcall: `__attribute__((fastcall))` — первые два аргумента в регистрах ECX, EDX.

* pascal: Прямого атрибута pascal в современном GCC обычно нет, так как исторический Pascal-call практически идентичен stdcall (но с другим порядком пуша аргументов — слева направо). Если вам нужно именно поведение Pascal, часто приходится использовать stdcall и вручную менять порядок аргументов в коде.

Пример для 32 бит:

```c++
int __attribute__((stdcall)) WriteSummaryText
    (
        const char *in, 
        char *out, 
        int size
    );
```
2. Архитектура x86_64 (64 бита)
   
В 64-битном Linux ситуация иная. Здесь существует только одно стандартное соглашение — System V AMD64 ABI.

Все специфические атрибуты вроде `stdcall`, `fastcall` или `cdecl` в 64-битном GCC для Linux игнорируются.

Компилятор выдаст предупреждение, но соберет код с использованием стандартного ABI (аргументы в регистрах `RDI`, `RSI`, `RDX`, `RCX`, `R8`, `R9`).

Единственное исключение — вы можете заставить функцию использовать Microsoft x64 ABI (если нужно вызвать её из кода, скомпилированного под Windows), используя `__attribute__((ms_abi))`.

3. "Заклинание" для управления

Если вы хотите, чтобы код был переносимым и работал как в Windows, так и в Linux, обычно используют макросы:

```c++
#ifdef _WIN32
  #define DLL_CALL __stdcall
#else
  // В Linux x86_64 это будет проигнорировано, в x86 — сработает
  #define DLL_CALL __attribute__((stdcall)) 
#endif

int DLL_CALL WriteSummaryText
    (
        const char *input, 
        char *output, 
        int size
    );
```
Итог:

* Если у вас `x86_64` (что скорее всего), управлять соглашением "внутри" Linux-мира нельзя — оно фиксировано.

* Если вам нужно имитировать старый Pascal/Stdcall для совместимости с внешним ПО на 32 битах, используйте `__attribute__((stdcall))`.

* Соглашение "Kernighan & Ritchie C call" — это и есть стандартный `cdecl`, который используется в GCC по умолчанию.

## Просмотр таблицы экспорта

В Microsoft Visual C++ есть утилита `dumpbin`, которую с помощью ключа `/exports` можно было попросить распечатать таблицу экспорта:

```bash
dumpbin /exports summary.dll
```
В Linux для этих целей есть несколько мощных инструментов. Если `dumpbin /exports` — это ваш привычный молоток, то в Linux у вас есть целый набор инструментов: от простых до «рентгеновских».

Вот основные способы проверить таблицу экспорта в `.so` файле:

1. `nm` — самый быстрый способ
   
Утилита `nm` выводит список всех символов. Чтобы увидеть именно динамические экспорты (то, что доступно извне), используйте флаг `-D`:

```bash
nm -D --defined-only libsummary.so
```
Что искать: Ищите букву `T` (от слова Text/Code) напротив имени функции. Если там `T`, значит функция экспортирована и определена.

Лайфхак: Добавьте флаг `-C`, и `nm` сам сделает demangle (превратит страшные C++ имена в читаемые).

2. readelf — «швейцарский нож» (аналог dumpbin)
   
Эта утилита дает самую детальную информацию о структуре ELF-файла. Для таблицы символов используйте:

```bash
readelf -Ws libsummary.so
```

Смотрите колонку `Name`. Обратите внимание на колонку `Vis` (Visibility): там должно быть `DEFAULT`. Колонка `Bind` должна содержать `GLOBAL`.

3. objdump — для фанатов ассемблера

Если вам нужно не только имя, но и адрес или даже код, используйте:

```bash
objdump -T libsummary.so
```

Флаг `-T` (от слова Dynamic Table) покажет ровно то, что увидит динамический линковщик при загрузке вашей библиотеки.

Подсказка по "манглингу": если вы все-таки используете C++ и забыли `extern "C"`, nm выдаст вам что-то вроде `_Z16WriteSummaryTextPKcPci`. Чтобы быстро понять, что это такое, не выходя из терминала, используйте упомянутую ранее утилиту `c++filt`:

```bash
echo _Z16WriteSummaryTextPKcPci | c++filt
# Вывод: WriteSummaryText(char const*, char*, int)
```

Итоговый рецепт для проверки: просто выполните `nm -D libsummary.so`. Если в списке есть ваше имя `WriteSummary` с пометкой `T` — поздравляю, ваше "заклинание" сработало!

Вот скрипт, который из вывода `nm -D` выбирает только имена функций, снабженных пометкой `T`:

```bash
#!/bin/bash

# Проверяем, передан ли файл
if [ -z "$1" ]; then
    echo "Использование: $0 path_to_so_file>
    exit 1
fi

# -D: только динамические символы
# -C: деманглинг имен (если это C++)
# --defined-only: убираем внешние зависимости (Undefined)
# awk: печатаем только если тип символа 'T' (код) или 'W' (слабый символ)
nm -DC --defined-only "$1" | awk '$2 ~ /[TW]/ { print $3 }'
```

Результат работы скрипта будет выглядеть примерно так:

```
WriteSummary
WriteSummaryText
_init
_fini
```

(Служебные функции вроде `_init` и `_fini` всегда присутствуют в ELF-файлах, их можно игнорировать).

## Version Script

Version Script — это текстовый файл (обычно с расширением `.map` или `.lds`), который дает вам полный контроль над тем, что «торчит» наружу из вашей библиотеки. Это гораздо мощнее, чем просто атрибуты в коде.
Вот структура правильного скрипта для вашей задачи:

1. Содержимое файла (например, `exports.map`)

```
LIB_SUMMARY_1.0 {
    global:
        # Экспортируем функцию под её родным именем
        WriteSummaryText;

        # Если вы создали alias в коде, его тоже надо упомянуть
        WriteSummary;

    local:
        # Скрываем всё остальное (аналог -fvisibility=hidden)
        *;
};
```

2. Как это работает (разбор по частям)

* `LIB_SUMMARY_1.0`: Это имя узла версии. В Linux принято версионировать символы. Если вы позже измените API, вы сможете создать узел LIB_SUMMARY_1.1, и старые программы не сломаются.
   
* `global::` Список имен, которые будут видны через nm -D. Здесь вы перечисляете функции, которые составляют ваш публичный интерфейс.
   
* `local: *;:` Самая важная строчка. Она говорит линкеру: «Все функции и переменные, которые я не упомянул в global, сделай приватными». Это очищает таблицу экспорта от мусора и служебных функций.

3. Как «скормить» это компилятору
   
При сборке вашей `.so` добавьте специальный флаг для линкера (`-Wl` означает «передать следующий параметр линкеру»):

```bash
gcc -shared -o libsummary.so summary.o -Wl,--version-script=exports.map
```

4. Продвинутый трюк: Алиас прямо в скрипте?

К сожалению, стандартный Version Script не умеет создавать алиасы (переименовывать A в B) сам по себе — он только управляет видимостью уже существующих имен.

Поэтому правильный алгоритм для Linux такой:

В коде используете `__attribute__((alias("WriteSummaryText"))) void WriteSummary();`.
   
В Version Script добавляете оба имени в секцию `global`.

## Makefile

Убедитесь, что в папке лежат:

* `summary.c` (с вашим кодом)

* `exports.map` (со списком функций)


```makefile
# Настройки компилятора
CC = gcc
CFLAGS = -fPIC -Wall -O2
LDFLAGS = -shared -Wl,--version-script=exports.map

# Имена файлов
TARGET = libsummary.so
SRC = summary.c
OBJ = $(SRC:.c=.o)

# Команда для проверки экспортов (наш bash-скрипт в одну строку)
CHECK_EXPORTS = nm -DC --defined-only $(TARGET) | awk '$$2 ~ /[TW]/ { print "  [EXPORTED] " $$3 }'

all: $(TARGET) check

# Сборка библиотеки
$(TARGET): $(OBJ)
	@echo "Linking $(TARGET)..."
	$(CC) $(LDFLAGS) -o $@ $^

# Компиляция объектного файла
%.o: %.c
	@echo "Compiling $<..."
	$(CC) $(CFLAGS) -c $< -o $@

# Финальный аккорд: проверка результата
check: $(TARGET)
	@echo "Verifying exported functions:"
	@$(CHECK_EXPORTS)

clean:
	rm -f $(OBJ) $(TARGET)

.PHONY: all clean check
```

Как это работает:

* `fPIC`: Обязательный флаг для создания разделяемых библиотек в Linux (Position Independent Code).

* `-Wl,--version-script`: Передает ваш файл со списком функций линкеру.

* `check`: После успешной сборки `Makefile` сам выполнит команду `nm` и красиво выведет список того, что в итоге "торчит" наружу.

* `@`: Символ перед командами подавляет вывод самой строки команды, оставляя в консоли только чистый результат.

Для запуска сборки достаточно написать в терминале `make`. Результат должен быть примерно таким:

```
Compiling summary.c...
Linking libsummary.so...
Verifying exported functions:
  [EXPORTED] WriteSummary
  [EXPORTED] WriteSummaryText
```
