# Запуск файлов на выполнение в Linux

## Как на языке C в среде Linux проверить существование файла с указанным именем?

В языке C для среды Linux существует несколько способов проверить существование файла. Выбор зависит от того, нужны ли вам права доступа к файлу или просто подтверждение его наличия на диске.

### 1. Использование функции `access` (рекомендуемый способ)

Это стандартный POSIX-метод. Флаг `F_OK` проверяет именно существование файла.

```c++
#include <unistd.h>
#include <stdio.h>

int main() {
    char *filename = "test.txt";

    // access возвращает 0, если файл существует
    if (access (filename, F_OK) == 0) {
        printf ("Файл существует\n");
    } else {
        printf ("Файл не найден\n");
    }
    return 0;
}
```

Плюсы: Быстрота (не считывает метаданные файла полностью) и простота кода.

### 2. Использование функции `stat`

Этот способ позволяет не только проверить наличие файла, но и получить информацию о нем (размер, тип, права).

```c++
#include <sys/stat.h>
#include <stdio.h>

int main() {
    struct stat buffer;
    char *filename = "test.txt";

    // stat возвращает 0 при успехе
    if (stat (filename, &buffer) == 0) {
        printf ("Файл существует\n");
    } else {
        printf ("Файл не существует\n");
    }
    return 0;
}
```

Важно: Чтобы отличить отсутствие файла от других ошибок (например, отсутствия прав на чтение папки), можно проверить глобальную переменную `errno` на значение `ENOENT`.

### 3. Использование `fopen` (стандарт ANSI C)

Самый переносимый способ, работающий не только в Linux, но и в Windows. Мы пытаемся открыть файл для чтения.

```c++
#include <stdio.h>

int main() {
    FILE *file;
    if ((file = fopen ("test.txt", "r"))) {
        fclose (file);
        printf ("Файл существует\n");
    } else {
        printf ("Файл не существует или недоступен\n");
    }
    return 0;
}
```

Минусы:

* Если у вас нет прав на чтение файла, `fopen` вернет `NULL`, даже если файл существует.
* Этот метод медленнее, так как создает полноценный системный поток ввода-вывода.

## Как я могу убедиться, что у меня есть права на запуск этого файла?

Раз вы уже используете `stat`, вам не нужно вызывать дополнительные функции. Вся необходимая информация о правах доступа уже содержится в поле `st_mode` структуры `stat`.

Однако есть важный нюанс: права в Linux делятся на три категории (владелец, группа и остальные). Чтобы понять, может ли текущий пользователь запустить файл, нужно сравнить его UID/GID с идентификаторами владельца файла.

## Способ 1: Анализ структуры `stat` (Вручную)

Этот метод надежен, так как работает с уже полученными данными. Вам нужно проверить биты исполнения (X) для соответствующей категории пользователей.

```c++
#include <sys/stat.h>
#include <unistd.h>
#include <stdio.h>

int can_execute (struct stat *st) {
    uid_t uid = geteuid(); // Получаем ID текущего пользователя

    // 1. Если мы — владелец файла
    if (uid == st->st_uid) {
        return st->st_mode & S_IXUSR;
    }

    // 2. Если мы входим в группу владельцев файла
    gid_t gid = getegid();
    if (gid == st->st_gid) {
        return st->st_mode & S_IXGRP;
    }

    // 3. Для всех остальных
    return st->st_mode & S_IXOTH;
}

// Пример использования:
// if (can_execute (&buffer)) { ... }
```


### Способ 2: Функция `access` (Самый простой)

Если вы не хотите вручную сравнивать UID и GID, проще всего после `stat` вызвать `access` с флагом `X_OK`. Система сама проверит все соответствия групп и прав за вас.

```c++
if (access (filename, X_OK) == 0) {
    printf ("У вас есть права на выполнение.\n");
} else {
    printf ("Запуск запрещен.\n");
}
```

### Важное дополнение: Проверка на «директорию»

Даже если у директории стоят биты исполнения (в Linux это означает право «войти» в папку), запустить её как программу нельзя. Поэтому перед проверкой прав обязательно убедитесь, что это обычный файл:

```c++
if (S_ISREG (buffer.st_mode)) {
    // Это файл, теперь можно проверять права на выполнение (X_OK)
} else {
    // Это директория, ссылка или устройство — запустить нельзя
}
```

Резюме: если данные `stat` уже на руках для каких-то других целей, используйте Способ 1. Если же вам нужно простое и быстрое решение «да/нет», Способ 2 (access) предпочтительнее.

## Допустим, ни для владельца файла, ни для группы бит разрешения запуска не установлен, но он установлен для "прочих". Позволит ли система мне запустить файл?

Короткий ответ: Нет, не позволит, если вы являетесь владельцем этого файла или входите в его группу.

В Linux действует принцип «первого совпадения» при проверке прав. Система проверяет категории по порядку: Владелец -> Группа -> Остальные. Как только она находит категорию, к которой вы относитесь, она берет права именно оттуда и дальше не смотрит.

# ## Как это работает на практике:

Представьте файл с правами `--- --- r-x` (в числовом виде 005):

1. Если вы владелец: Система видит, что для владельца прав нет (`---`). Она не будет смотреть на права группы или «прочих». Вы получите ошибку Permission denied.

2. Если вы не владелец, но входите в группу: Система увидит права группы (`---`) и запретит запуск.

3. Если вы «никто» для этого файла (не владелец и не в группе): Только тогда система дойдет до секции «others» и увидит там `r-`x. В этом случае запуск разрешится.

### Почему так сделано?

Это позволяет реализовывать специфические запреты. Например, вы можете разрешить доступ к файлу всем сотрудникам компании, кроме одного конкретного человека, сделав его владельцем файла и установив ему права `000`.

### Как это проверить программно?

Если вы используете функцию `access (filename, X_OK)`, она отработает именно по этой системной логике — она учтет, кем вы являетесь по отношению к файлу, и выдаст честный результат.

А если вы используете тот код с ручной проверкой через `stat`, который мы обсуждали выше, то он тоже учитывает эту иерархию:

```c++
if (uid == st->st_uid) {
    return st->st_mode & S_IXUSR; // Только права владельца!
} else if (is_in_group (st->st_gid)) {
    return st->st_mode & S_IXGRP; // Только права группы!
} else {
    return st->st_mode & S_IXOTH; // Только тогда — для всех остальных.
```

## Требуется ли `--x`, чтобы загрузить динамическую библиотеку с помощью `dlopen`?

Короткий ответ: Нет, права на исполнение (`--x`) для самой библиотеки не требуются.
Для успешной работы `dlopen` достаточно, чтобы у файла библиотеки были права на чтение (`-r-`).

### Почему так происходит?

Когда вы вызываете `dlopen`, операционная система не «запускает» библиотеку как самостоятельную программу. Происходит следующее:

1. Ядро или динамический линковщик открывает файл на чтение.

2. Содержимое файла отображается в адресное пространство вашего процесса с помощью системного вызова `mmap`.

3. При отображении сегментов кода (text segment) линковщик сам запрашивает у системы права PROT_EXEC для этих страниц памяти, чтобы процессор мог выполнять инструкции внутри них.

Право `--x` в файловой системе Linux нужно только для того, чтобы ядро разрешило использовать файл в качестве точки входа для нового процесса (через семейство вызовов 
`execve`), то есть для запуска бинарников или скриптов.

### На что действительно стоит обратить внимание:

* Права на каталоги: У пользователя, от имени которого работает приложение, должны быть права на выполнение (`--x`) для всех родительских папок, где лежит библиотека. В контексте директорий +x означает право «войти» в папку и обратиться к объектам внутри неё.

* SELinux / AppArmor: Если у вас включены системы принудительного контроля доступа, они могут блокировать загрузку библиотек из нестандартных путей (например, из `/tmp` или домашних папок), даже если права доступа в файловой системе выставлены верно.

* Флаг `noexec`: Если раздел диска, на котором лежит `libllm.so`, смонтирован с опцией `noexec`, вызов `dlopen` завершится ошибкой, так как система запрещает исполнение любого кода с этого носителя.

## Если я использую `exec()`, система проверит бит `--x`?

С функцией `exec()` (и всем её семейством: `execl`, `execv` и т. д.) ситуация практически такая же, как и с `system`: права на исполнение (`--x`) обязательны. Вот как это работает на уровне системы:

### 1. Как работает `exec()`

В отличие от `dlopen`, которая просто подгружает код в уже работающий процесс, функции семейства `exec()` полностью заменяют текущий процесс новым образом. Когда вы вызываете `exec()`, управление передается ядру (системный вызов `execve`). Ядро Linux перед запуском проверяет файл по нескольким критериям:

* Является ли файл бинарным (ELF) или скриптом (с шебангом `#!`).

* Есть ли у текущего пользователя права на исполнение (`--x`) этого файла.

Если бита `--x` нет, `exec()` вернет -1, а в переменной `errno` будет установлена ошибка `EACCES` (Permission denied).

### 2. Разница между `system()` и `exec()`

Хотя обоим нужны права `--x`, логика проверки немного отличается:

* `system("my_prog")`: Сначала запускает `/bin/sh`. Именно оболочка (shell) пытается найти и запустить ваш файл. Если прав нет, shell выдаст ошибку (например, "Permission denied") и вернет статус завершения.

* `execvp("./my_prog", ...)`: Ваше приложение напрямую просит ядро заменить себя на `my_prog`. Проверку прав выполняет само ядро.

### Нюанс со скриптами

Если вы пытаетесь сделать `exec()` для текстового скрипта (например, на Python или Bash):

1. С правами `--x`: Ядро видит шебанг `#!/usr/bin/python3`, само запускает интерпретатор и передает ему ваш скрипт.

2. Без прав `--x`: `exec()` выдаст ошибку. Но такой скрипт можно запустить в обход, вызвав интерпретатор напрямую: `execl ("/usr/bin/python3", "python3", "my_script.py", NULL)`. В этом случае `--x` нужен интерпретатору, а для самого скрипта достаточно прав на чтение (`-r-`).

Итог: для бинарного файла (скомпилированного приложения), который вы хотите запустить через 
`exec()`, бит `--x` критически необходим.

## Запуск программы через `fork` + `exec`

Для Linux-приложений этот способ считается предпочтительным. Вы напрямую просите ядро создать процесс и заменить его на нужную программу.

Почему это лучше для вас:

* Безопасность: Аргументы передаются как массив, оболочка не вызывается, интерпретация символов невозможна.

* Контроль: Вы точно знаете, запустилась ли программа, и можете получить её точный код возврата.

Пример реализации:

```c++
#include <sys/types.h>
#include <sys/wait.h>
#include <unistd.h>
#include <stdlib.h>
#include <stdio.h>
#include <errno.h>

int run_external_program (const char *in, const char *out) {
    pid_t pid = fork();

    if (pid == -1) {
        perror ("fork failed");
        return -1;
    }

    if (pid == 0) {
        // Это дочерний процесс
        // Вызываем программу. Аргументы: путь, arg0, arg1, arg2, NULL
        execlp ("external-program", "external-program", in, out, (char *) NULL);
        
        // Если execlp вернул управление — значит произошла ошибка
        perror ("execlp failed");
        exit (EXIT_FAILURE);
        
    } else {
        // Это родительский процесс (ваше расширение)
        int status;
        // Ждем завершения именно нашей программы
        if (waitpid (pid, &status, 0) == -1) {
            perror ("Ошибка при ожидании процесса (waitpid)");
            return -1;
        }

        if (WIFEXITED (status)) {
            int exit_code = WEXITSTATUS (status);
            if (exit_code != 0) {
                fprintf (stderr, "Программа завершилась с ошибкой. Код: %d\n", exit_code);
                return exit_code;
            }
            
            return 0; // Все отлично
        } else if (WIFSIGNALED (status)) {
            fprintf (stderr, "Программа была убита сигналом: %d\n", WTERMSIG (status));
            return -1;
        }
    }
    return -1;
}
```
